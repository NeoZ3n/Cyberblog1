1) The moving parts of Entra authentication
Identities and clients. Entra authenticates users (employees/guests), service principals (applications), and managed identities (Azure resources). Apps use standard OAuth 2.0/OIDC/SAML/WS‑Fed flows, commonly via Microsoft Authentication Libraries (MSAL). [learn.microsoft.com], [learn.microsoft.com]
Authentication methods (human sign‑ins). Modern, phishing‑resistant options include passkeys (FIDO2) and Windows Hello for Business, alongside certificate‑based authentication (CBA); Microsoft Authenticator push uses number matching by default; Temporary Access Pass (TAP) bootstraps passwordless or recovery; SMS/voice/OATH remain available but are lower assurance. [learn.microsoft.com], [learn.microsoft.com], [learn.microsoft.com], [learn.microsoft.com], [learn.microsoft.com], [learn.microsoft.com]
Workload authentication (non‑human). Apps authenticate with client credentials (secret or, preferably, certificate) or through workload identity federation that trusts an external OIDC issuer (e.g., GitHub Actions, Kubernetes), eliminating long‑lived secrets. Azure managed identities are platform‑managed service principals tied to resources. [learn.microsoft.com], [github.com], [learn.microsoft.com], [learn.microsoft.com]
Device SSO & PRT. During device registration/join, Windows (and other platforms) obtain a Primary Refresh Token (PRT) for SSO; PRT is device‑bound and TPM‑protected when available, and later redeemed silently by the broker (WAM). [learn.microsoft.com]
Where the truth lives (logs). Sign‑ins expose who/how/what plus RequestId/CorrelationId and per‑step AuthenticationDetails in the Entra portal, Graph, and Azure Monitor (Log Analytics/Sentinel). These are your primary artifacts for detection & DFIR. [learn.microsoft.com], [learn.microsoft.com]

2) Anatomy of a sign‑in: phases & controls
Plain Text[Client] ─► [Entra authorize/token endpoints]   1) Pre-auth (tenant checks, policy routing)   2) Primary auth (passwordless/password/cert)   3) Step-up/MFA (Conditional Access as needed)   4) Token issuance (ID/Access/Refresh or PRT)       ▼[Resource/API] (enforces CAE; may challenge)Afficher plus de lignes
Session & reauthentication. Sign‑in frequency (SIF) controls browser/app session reauth cadence; you can require reauth “every time” for sensitive apps/actions (e.g., PIM activation) to reduce token‑theft blast radius. [learn.microsoft.com], [o365reports.com]
Token lifetimes. Access tokens are short‑lived (~1h typical) and aren’t revoked mid‑lifetime; refresh tokens last longer (often 90 days; 24h for SPAs) and can be revoked; certain lifetimes are configurable with caveats. [learn.microsoft.com], [docs.azure.cn]
Continuous Access Evaluation (CAE). Supported workloads (Graph/EXO/SPO/Teams) receive near‑real‑time signals to stop accepting a token after critical events (account disable, password reset, location change), forcing the client to reauthenticate via a claims challenge. Is CAE token is visible in sign‑in logs and a dedicated workbook. [learn.microsoft.com], [learn.microsoft.com]
Device SSO & reauth. With a PRT, apps can silently acquire tokens; Conditional Access may still demand step‑up. WHfB gestures unlock local credentials but do not preclude server‑side MFA requirements. [learn.microsoft.com], [learn.microsoft.com]

Typical for modern apps; the interactive step appears in SigninLogs, subsequent silent refreshes often surface in AADNonInteractiveUserSignInLogs. [learn.microsoft.com], [learn.microsoft.com], [docs.azure.cn]
3.2 Windows device SSO and PRT
Windows device registration/join issues a PRT; the broker (WAM) redeems it to acquire app tokens silently. Conditional Access may still require MFA. [learn.microsoft.com]
3.3 Device Code Flow (CLI/IoT/shared screens)
An input‑constrained device displays a user_code; the user completes auth on another device; tokens are issued to the original device. Expect AuthenticationProtocol = deviceCode in logs. [learn.microsoft.com]
3.4 Service principals (client credentials)
App authenticates as itself with a secret/certificate (prefer certificates). No refresh tokens in this flow; the app requests new access tokens directly. Events land in AADServicePrincipalSignInLogs. [learn.microsoft.com], [learn.microsoft.com]
3.5 Managed identities
Azure assigns an identity to a resource; the workload exchanges an IMDS token for an Entra access token. Events appear in AADManagedIdentitySignInLogs. [learn.microsoft.com], [learn.microsoft.com]
3.6 External IdP federation (SAML/WS‑Fed)
For federated external domains, Entra redirects to the partner IdP and, on success, issues downstream tokens. Federation details surface in processing fields and applied CA policies. [learn.microsoft.com]

4) Authentication methods: what actually happens

Passkeys (FIDO2) & Windows Hello for Business. Device‑bound keys sign a server challenge (WebAuthn/CTAP). These are phishing‑resistant and can satisfy MFA/strength requirements depending on policy. [learn.microsoft.com], [learn.microsoft.com]
Authenticator push with number matching. The sign‑in page shows a number the user must enter into Authenticator; push‑fatigue mitigation and enabled by default for push. [learn.microsoft.com], [learn.microsoft.com]
Certificate‑based authentication (CBA). Entra validates an X.509 against configured CAs/CRLs and user mapping; when configured it replaces passwords for primary auth. Ensure CRL endpoints are reachable. [learn.microsoft.com]
Temporary Access Pass (TAP). Time‑limited passcode (single or multi‑use) to bootstrap passwordless or recover when no strong method is available; governed by the Authentication Methods policy. [learn.microsoft.com]
SMS/voice/OATH. Supported, but consider phishing risk versus phishing‑resistant methods. Use Authentication Strengths in CA to enforce preferred methods. [learn.microsoft.com]


5) Where each step is logged (and how to read it)
Core tables (Azure Monitor / Sentinel):

SigninLogs — interactive user sign‑ins (primary + step‑up). Fields to know: AuthenticationDetails (array of steps), AuthenticationRequirement, AuthenticationMethodsUsed, AuthenticationProcessingDetails, ClientAppUsed, ConditionalAccessPolicies, RequestId, CorrelationId. [learn.microsoft.com]
AADNonInteractiveUserSignInLogs — background refreshes, silent SSO, token exchanges on behalf of the user; events are often grouped for digestibility. [learn.microsoft.com], [docs.azure.cn]
AADServicePrincipalSignInLogs — app‑only sign‑ins; check ClientCredentialType (secret/certificate/federated credential) and CA outcome. [learn.microsoft.com]
AADManagedIdentitySignInLogs — managed identity sign‑ins with MSI details and any federated credential IDs. [learn.microsoft.com]
AADUserRiskEvents / AADRiskyUsers — Identity Protection detections correlated to sign‑ins via RequestId. [learn.microsoft.com]
AuditLogs — admin/configuration changes that can explain auth behavior changes. [learn.microsoft.com]

Portal vantage point. The Entra Sign‑in logs blade shows “Basic info”, “Conditional Access”, and per‑step Authentication details with unique IDs; the built‑in Sign‑in Diagnostics can accelerate troubleshooting. [learn.microsoft.com]
CAE visibility. Filter by Is CAE token in sign‑ins, or use the CAE insights workbook to view token challenges and IP mismatches across the tenant. [learn.microsoft.com]

6) Log visibility by authentication method (field notes)

Use these to sanity‑check detections and guide hunting. Each scenario below can be reproduced in a lab and verified in your tenant.

A) Password + Authenticator (number matching)

Expect in SigninLogs:
AuthenticationRequirement = "multiFactorAuthentication"; AuthenticationDetails[0].authenticationMethod = "Password" then [1].authenticationMethod = "Microsoft Authenticator"/Push (labels vary); CA policy list in ConditionalAccessPolicies; device/app info. OTP codes/number values are never logged. [learn.microsoft.com], [learn.microsoft.com]

B) Passkey (FIDO2) passwordless

Expect: AuthenticationMethodsUsed includes FIDO; AuthenticationDetails[0].authenticationMethod = "FIDO"; AuthenticationRequirement equals singleFactorAuthentication or multiFactorAuthentication depending on your Authentication Strength policy. [learn.microsoft.com]

C) Certificate‑based authentication (CBA)

Expect: AuthenticationDetails[0] shows certificate‑based method; AuthenticationProcessingDetails contains certificate processing info; failures reveal CA/CRL/mapping issues. [learn.microsoft.com]

D) TAP bootstrap to register passkey/WHfB

Expect: AuthenticationMethodsUsed contains Temporary Access Pass; after registration, future sign‑ins show the new passwordless method as primary. [learn.microsoft.com]

E) Device SSO/PRT (WHfB unlock + silent token)

Expect: The initial interactive sign‑in in SigninLogs; subsequent silent refreshes in AADNonInteractiveUserSignInLogs (often grouped); device information appears in DeviceDetail. [learn.microsoft.com], [docs.azure.cn]

F) Service principal (certificate)

Expect: AADServicePrincipalSignInLogs with ClientCredentialType = "certificate", plus AppId/ServicePrincipalId, target resource, IP, and CA outcome. [learn.microsoft.com]

G) Managed identity

Expect: AADManagedIdentitySignInLogs with ManagedServiceIdentity payload (user‑ or system‑assigned), resource, and optional FederatedCredentialId when configured. [learn.microsoft.com]

H) Device code flow

Expect: AuthenticationProtocol = "deviceCode" on the interactive record for the approving user; useful to flag unusual device code usage patterns. [learn.microsoft.com]

I) SAML/WS‑Fed federation

Expect: AuthenticationProcessingDetails indicates external IdP; errors often reflect issuer/claims mismatches; CA evaluation applies post‑federation. [learn.microsoft.com]


7) Session management & revocation in practice

Use Sign‑in frequency (SIF) to bound session life; add reauth “every time” for sensitive workloads (e.g., PIM activation, admin portals) to inject guaranteed checks despite valid tokens. [learn.microsoft.com], [o365reports.com]
Understand tokens: access tokens aren’t revoked mid‑lifetime; refresh tokens can be revoked; CAE provides near‑real‑time enforcement by cooperating resources. Validate CAE usage via the workbook and the Is CAE token filter. [learn.microsoft.com], [docs.azure.cn], [learn.microsoft.com], [learn.microsoft.com]


8) Field guide: columns you’ll actually use

RequestId + CorrelationId — glue between token, API, and sign‑ins; extract from token or app logs to line up records precisely. [learn.microsoft.com]
AuthenticationDetails — the per‑step gold (method, result, time). Not all scenarios populate equally (e.g., some B2B cases). [learn.microsoft.com]
AuthenticationProcessingDetails — federation/processing/broker hints (PTA/PHS, server names). [learn.microsoft.com]
ConditionalAccessPolicies / ConditionalAccessStatus — which CA policies applied and whether they blocked/required MFA. [learn.microsoft.com]
Non‑interactive tables — query both interactive and non‑interactive to understand the full lifecycle of a user’s session. [isroot.nl]
